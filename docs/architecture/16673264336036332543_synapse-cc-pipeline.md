# synapse-cc: The Client Generation Pipeline

## Overview

synapse-cc (Synapse Compiler Collection) is the orchestrator that automates the full pipeline from live Plexus backend schema to compiled, typed client libraries. It sits at the top of the toolchain:

```
Plexus Backend (Rust)
    │
    │  WebSocket JSON-RPC
    ▼
plexus-synapse (Haskell, in-process)
    │
    │  IR JSON (types, methods, refs)
    ▼
hub-codegen (Rust, subprocess)
    │
    │  Generated source files (JSON envelope)
    ▼
synapse-cc (Haskell, orchestrator)
    │
    │  Three-way merge, deps, build, test
    ▼
Typed Client Library (TypeScript, etc.)
```

## Design Principles

### hub-codegen is stateless

hub-codegen is a pure code generator: IR JSON in, generated files JSON out. It has no knowledge of package managers, caches, or filesystems (when called with `--output-format json`). This makes it deterministic and testable in isolation.

### synapse-cc owns orchestration

Everything outside of pure code generation belongs to synapse-cc:

- **Cache management** — two-level cache (IR + code) with per-plugin invalidation
- **Three-way merge** — SHA-256 hash comparison preserves user edits across regeneration
- **Dependency management** — detects package manager (bun/pnpm/yarn/npm), installs packages
- **Build & test** — TypeScript compilation, test execution (standalone mode)
- **Watch mode** — polls backend for schema changes, rebuilds incrementally
- **Mode detection** — standalone (synapse-cc owns the full project) vs integration (host project owns build)

### plexus-synapse is a library, not a subprocess

synapse-cc imports plexus-synapse as a Haskell library and calls IR generation in-process. This avoids subprocess overhead and shares the Haskell runtime. The same IR builder that powers `synapse --emit-ir` on the CLI is called directly from synapse-cc's pipeline.

## Pipeline Stages

### 1. Cache Validation

Reads cached manifests from `~/.cache/plexus-codegen/synapse/`. Checks:
- Tool versions (synapse, hub-codegen) — version change invalidates everything
- Per-plugin schema hashes — only changed plugins trigger regeneration
- synapse.lock file — tracks what was generated and when

### 2. IR Generation (~77ms for 39 plugins)

Calls plexus-synapse library directly:
- Connects to backend at configured host:port
- Fetches schemas with parallel plugin discovery
- Builds IR with type deduplication (content-hash based)
- Writes IR cache: `~/.cache/plexus-codegen/synapse/ir/{backend}/ir.json`

The IR contains:
- `irTypes` — all types deduplicated across namespaces, with kind info (struct, enum, string enum, alias, primitive)
- `irMethods` — all methods with typed params, return types, streaming inference, bidirectional channel types
- `irPlugins` — plugin → method name mapping
- `irPluginHashes` — per-plugin content hashes for granular invalidation
- `irMetadata` — toolchain versions, generation timestamp

### 3. Code Generation (~31ms)

Runs hub-codegen as a subprocess:
```
hub-codegen --target typescript --output-format json --transport ws ir.json
```

hub-codegen returns a JSON envelope:
```json
{
  "files": { "src/echo.ts": "...", "src/client.ts": "..." },
  "fileHashes": { "src/echo.ts": "sha256:...", ... },
  "dependencies": { "ws": "^8.0.0" },
  "devDependencies": { "typescript": "^5.0.0" },
  "warnings": [],
  "hubCodegenVersion": "0.3.0"
}
```

## Configuration

synapse-cc is driven by a `synapse.config.json` file in the project root, created via `synapse-cc init`:

```json
{
  "schema": "1.0",
  "language": "typescript",
  "backend": "substrate",
  "url": "ws://127.0.0.1:4444",
  "packageManager": "bun",
  "watch": {
    "pollInterval": 1000,
    "hotReload": false
  },
  "targets": {
    "client": {
      "generate": ["transport", "rpc", "plugins"],
      "transport": "ws",
      "outputDir": "src/lib/plexus"
    }
  }
}
```

### Top-Level Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `schema` | string | Yes | Config schema version (`"1.0"`) |
| `language` | string | Yes | Target language: `"typescript"`, `"python"`, or `"rust"` |
| `backend` | string | No* | Backend identifier (e.g., `"substrate"`). Required unless all targets are transport-only |
| `url` | string | No* | WebSocket URL to backend. Required unless all targets are transport-only |
| `packageManager` | string | No | Package manager (`"bun"`, `"pnpm"`, `"yarn"`, `"npm"`). Default: `"bun"` |
| `watch` | object | No | Watch mode config: `pollInterval` (ms), `hotReload` (reload config on change) |
| `targets` | object | Yes | Named build targets (at least one required) |

### Target Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `generate` | array | Yes | What to generate: `["transport"]`, `["rpc"]`, `["plugins"]`, or `["all"]` |
| `transport` | string | Yes | `"ws"` (Node.js, uses `ws` package) or `"browser"` (native WebSocket, for Tauri/WebView) |
| `outputDir` | string | Yes | Where generated code is written |
| `backend` | string | No | Per-target backend override |
| `url` | string | No | Per-target URL override |
| `plugins` | array | No | Plugin name prefixes to include (omit for all) |
| `smokePath` | string | No | Relative path to transport module for smoke tests |

### Auto-Detection

`synapse-cc init` detects the project type and sets transport accordingly:

1. **Tauri** (`src-tauri/` or `tauri.conf.json`) → `"browser"` transport
2. **Vite** (`vite.config.ts/js/mts/mjs`) → `"browser"` transport
3. **Next.js** (`next.config.js/ts/mjs`) → `"browser"` transport
4. **Node.js** (`package.json`) → `"ws"` transport
5. **Fallback** → `"ws"` transport

### Multiple Targets

Multiple targets support different environments from the same backend:

```json
{
  "targets": {
    "server": {
      "generate": ["all"],
      "transport": "ws",
      "outputDir": "src/lib/plexus"
    },
    "browser": {
      "generate": ["all"],
      "transport": "browser",
      "outputDir": "src/lib/plexus-browser"
    }
  }
}
```

### 4. Mode Detection

- **Standalone**: No existing package.json or tsconfig.json — synapse-cc scaffolds everything (package.json, tsconfig.json, directory structure)
- **Integration**: Host project exists — synapse-cc only writes generated files into the configured output directory, leaves build config to the host

### 5. Three-Way Merge

For each generated file, compares three versions:
1. **Previous generated** (from cache hash)
2. **Current on disk** (what the user may have edited)
3. **Newly generated** (from hub-codegen)

Decision matrix:
- Disk matches previous generated → overwrite with new (no user edits)
- Disk differs from previous generated → preserve user edits (skip or merge)
- No previous generated → write new file
- New matches previous → skip (nothing changed)

SHA-256 hashes are stored in the code cache manifest for comparison.

### 6. Dependencies

After merge, synapse-cc:
- Detects the project's package manager (checks for bun.lockb, pnpm-lock.yaml, yarn.lock, package-lock.json)
- Adds required dependencies from hub-codegen's output
- Runs install (`bun install`, `pnpm install`, etc.)

### 7. Build & Test (standalone mode only)

- Runs `tsc` for TypeScript compilation
- Runs test suite if present (`bun test`, etc.)

### 8. Cache Write

Updates:
- IR manifest with per-plugin hashes
- Code manifest with file hashes and tool versions
- `synapse.lock` with generation metadata

## Watch Mode

```bash
synapse-cc watch substrate --plugins echo,cone
```

Polls the backend for schema changes at a configurable interval. When a plugin's schema hash changes:
1. Regenerate IR for changed plugins only
2. Re-run hub-codegen
3. Three-way merge
4. Rebuild if standalone mode

Per-plugin granularity means changing the `echo` activation doesn't regenerate code for `cone`.

## Performance

Full force rebuild (39 plugins):
```
Total:           ~124ms
  IR generation:   77ms (62%)
  Code generation: 31ms (25%)
  Merge + cache:   <5ms  (4%)
  Startup:         11ms  (9%)

Cache hit:         ~32ms (manifest reads only)
```

The bottleneck is network latency + JSON parsing during IR generation, which is unavoidable for a live-schema system.

## File Layout

```
synapse-cc/
  app/Main.hs                  # CLI entry point (init, build, watch)
  src/SynapseCC/
    Pipeline.hs                # Full orchestration (stages 1-8)
    Types.hs                   # Config, Target, Options, errors
    Discover.hs                # Tool discovery (synapse, hub-codegen)
    Cache.hs                   # Version-aware two-level cache
    Merge.hs                   # Three-way SHA-256 merge
    Watch.hs                   # Watch mode polling

hub-codegen/                   # Stateless Rust code generator (sibling)
```

## Relationship to Synapse

synapse-cc depends on plexus-synapse as a library. The key shared components:

- **IR types** (`Synapse.IR.Types`) — IRData, TypeDef, MethodDef, TypeRef
- **IR builder** (`Synapse.IR.Builder`) — schema → IR transformation
- **Transport** (`Synapse.Transport`) — WebSocket connection to backend
- **Backend discovery** (`Synapse.Backend.Discovery`) — finding backends via registry

synapse-cc does NOT shell out to the `synapse` binary. It calls the library functions directly, which means IR generation shares the same code path as `synapse --emit-ir` but without process startup overhead.
