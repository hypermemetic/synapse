## Issue 1: Array Comma-Splitting Bug ✅ FIXED

### Problem

When passing array values via CLI with shell quoting like:
```bash
synapse foo --multi "a,b,c","d,e,f"
```

The shell correctly splits this into two arguments: `["a,b,c", "d,e,f"]`. However, synapse then re-split each value on commas, producing six values instead of two.

### Root Cause

**File**: `src/Synapse/CLI/Parse.hs:205-211`

```haskell
RefArray innerType ->
    let rawValues = [val | ("", val) <- kvs]
        values = concatMap splitCommas rawValues  -- BUG: splits again!
    in if null values
       then Left $ InvalidValue (pdName param) "array requires at least one value"
       else Right $ Array $ V.fromList $ map (buildTypedValue ir innerType) values
```

The `splitCommas` function was automatically splitting comma-separated values, which violated the principle of respecting shell quoting.

### Solution

Removed automatic comma-splitting. Values are now used as-is from the shell:

```haskell
RefArray innerType ->
    let values = [val | ("", val) <- kvs]  -- Direct use, no splitting
    in if null values
       then Left $ InvalidValue (pdName param) "array requires at least one value"
       else Right $ Array $ V.fromList $ map (buildTypedValue ir innerType) values
```

Also removed the now-unused `splitCommas` helper function.

### Impact

Array values now correctly respect shell quoting. Users can:
- Use shell separation: `--multi "a,b,c" "d,e,f"` → 2 values
- Repeat flags: `--multi "a,b,c" --multi "d,e,f"` → 2 values
- Include commas in values: `--text "hello, world"` → 1 value with comma

### Tests

Existing tests pass:
- `parse-test`: ✅ All array handling tests pass
- `path-normalization-test`: ✅ All tests pass

## Issue 2: --path Requirement ✅ FIXED (Previously)

### Documented Issue

**Source**: `hyperforge/docs/plans/LFORGE2-1.md:291`

> All single-repo commands require `--path` (synapse limitation - can't use implicit CWD)

### Actual State: RESOLVED

This comment is **outdated**. Synapse now has smart defaults for path parameters.

**File**: `src/Synapse/CLI/Transform.hs:366-370`

```haskell
getSmartDefault :: TransformEnv -> ParamDef -> IO (Maybe Text)
getSmartDefault env param
  -- Path parameters: use current working directory
  | isPathParamByType param =
      pure $ Just (T.pack $ teCwd env)
```

When a path parameter is required but not provided, synapse automatically injects the current working directory as the default. This happens during the `injectSmartDefaults` transformation phase.

### Impact

Commands can now omit `--path` and it will default to CWD:
```bash
# Old (still works)
synapse backend method --path /some/path

# New (also works)
cd /some/path
synapse backend method  # Uses CWD automatically
```

### Action Items

- [ ] Update hyperforge documentation to remove this limitation note
- [ ] Document smart defaults feature in synapse docs

## Issue 5: Float Timestamp in plexus-rpc-ts ❌ STILL BROKEN

### Problem

**File**: `plexus-rpc-ts/src/server.ts:68`

```typescript
function meta(): StreamMetadata {
  return {
    provenance: [name],
    plexusHash: rootHash,
    timestamp: Date.now() / 1000  // ❌ Produces float: 1773801011.821
  }
}
```

### Why This Breaks

Haskell's `StreamMetadata` type has `metaTimestamp :: Integer`.

Aeson's `FromJSON Integer` parser:
- Integer JSON number → ✅ Parses successfully
- Float JSON number → ❌ **Silent parse failure**

When parsing fails:
1. Haskell prints `"Failed to parse stream item"` to stderr
2. Loops forever waiting for `StreamDone` that was already sent
3. User sees synapse hang with no clear error

### Impact

**Critical**: Affects ALL stream items (data, error, done, progress) since they all use `meta()`.

Any synapse client connecting to a plexus-rpc-ts server will **hang silently** on every RPC call.

### Solution

```typescript
function meta(): StreamMetadata {
  return {
    provenance: [name],
    plexusHash: rootHash,
    timestamp: Math.floor(Date.now() / 1000)  // ✅ Integer: 1773801011
  }
}
```

### Documentation vs Reality

The fix is **documented** in:
- `plexus-gamma/docs/protocol/issues/synapse/ts-server.md:70`
- `plexus-gamma/docs/protocol/guides/synapse-compatible-ts-server.md:99`

But **not implemented** in `plexus-rpc-ts/src/server.ts:68`.

### Action Items

- [ ] Fix `plexus-rpc-ts/src/server.ts:68` to use `Math.floor(Date.now() / 1000)`
- [ ] Add integration test: synapse → plexus-rpc-ts server
- [ ] Verify plexus-gamma/src/lib/useBridge.ts:17 (also has `Date.now() / 1000`)

### Issue 6.2: _info Subscription Protocol ⚠️ DUAL IMPLEMENTATION

**Problem**: plexus-rpc-ts has TWO `_info` handlers:

**Direct response** (lines 212-220):
```typescript
if (msg.method === '_info') {
  ws.send(JSON.stringify({
    jsonrpc: '2.0', id: msg.id,
    result: { backend: name, version: rootSchema?.version ?? '0.1.0' },
  }))
  return
}
```

**Subscription-based** (lines 116-122):
```typescript
if (innerMethod === '_info') {
  sendData(ws, subId, { backend: name, version: rootSchema?.version ?? '0.1.0' })
  sendDone(ws, subId)
  return
}
```

The subscription handler is only reached when calling `{backend}.call` with `method: '_info'`. Direct `_info` calls use the problematic direct response.

**Impact**: Synapse expects subscription protocol for ALL calls, including `_info`. Direct response causes:
1. Haskell parses response as `SubscriptionId`
2. Registers subscription queue under garbage ID
3. Waits forever for notifications

**Status**: This might work if synapse only calls `{backend}.call` and never sends direct `_info` method. Needs verification.

**Action Items**:
- [ ] Test how synapse actually calls `_info` (direct vs via .call)
- [ ] Remove direct `_info` handler if it causes issues
- [ ] Add integration test
